## 세션 관리

### 1. 통신 구간에서의 평문 노출 (가장 큰 위험)

현재 구조에서는 브라우저가 비밀번호를 암호화하지 않고 서버로 보냅니다. 만약 이 과정이 일반적인 HTTP 환경에서 이루어진다면, 중간자 공격(Man-in-the-Middle)을 통해 해커가 네트워크 패킷을 가로채 사용자 비밀번호를 그대로 읽을 수 있습니다.

- **해결책**: 반드시 **HTTPS(SSL/TLS)**를 적용해야 합니다. HTTPS는 브라우저와 서버 사이의 통신 전체를 암호화하므로, 전송 중인 평문 비밀번호가 외부로 노출되지 않도록 보호합니다.

### 2. 클라이언트 측 암호화의 한계

"브라우저에서 미리 암호화해서 보내면 더 안전하지 않을까?"라고 생각할 수 있지만, 프론트엔드에서의 암호화는 생각보다 보안 효과가 크지 않습니다.

- **재전송 공격**: 브라우저에서 비밀번호를 해싱해서 보내더라도, 해커가 그 해시값 자체를 가로채서 서버에 보내면 서버는 이를 정상적인 비밀번호로 인식해 로그인을 허용할 수 있습니다.
- **결론**: 결국 서버에 도착한 데이터를 어떻게 처리하느냐가 더 중요하며, 전송 구간의 보안은 HTTPS에 맡기는 것이 웹 표준 방식입니다.

### 3. 서버 측 세션 관리의 취약점

현재 방식은 Flask의 `session` 객체를 사용하며, 이는 브라우저의 쿠키에 세션 ID를 저장하는 방식입니다.

- **세션 하이재킹 (Session Hijacking)**: 해커가 사용자의 쿠키를 탈취하면 해당 사용자로 위장하여 시스템에 접근할 수 있습니다.
- **보완**: 쿠키 설정 시 `HttpOnly`(자바스크립트로 쿠키 접근 불가), `Secure`(HTTPS에서만 전송), `SameSite`(교차 사이트 요청 방지) 옵션을 활성화해야 합니다.

- **CSRF (Cross-Site Request Forgery)**: 사용자가 로그인된 상태에서 악성 사이트를 방문했을 때, 본인도 모르게 게시글을 쓰거나 삭제하는 요청이 서버로 보내질 수 있습니다.
- **보완**: Flask-WTF 같은 라이브러리를 사용해 **CSRF 토큰**을 검증하는 로직을 추가해야 합니다.

### 4. 서버 측 비밀번호 해싱 방식

현재 `app.py`는 `generate_password_hash`를 사용하고 있습니다. 이는 단순한 암호화보다 훨씬 안전한 **단방향 해싱** 방식입니다.

- **안전한 점**: 서버 DB가 해킹당하더라도 해커는 복호화된 원래 비밀번호를 알 수 없습니다.
- **보완**: 해싱 알고리즘이 `pbkdf2:sha256`이나 `bcrypt` 같은 강력한 알고리즘을 사용하는지 확인하고, 작업 계수(Work Factor)를 높여 무차별 대입 공격(Brute-force)을 방어해야 합니다.

## 네트워크 보안과 암호화 시스템

### 1. 대칭키 (Symmetric Key) vs 비대칭키 (Asymmetric Key)

이 두 개념은 **암호화와 복호화에 사용하는 키의 일치 여부**에 따른 분류입니다.

#### 대칭키 (Symmetric Key)

- **개념:** 암호화할 때 사용하는 키와 복호화할 때 사용하는 키가 동일한 방식입니다.
- **특징:** 알고리즘이 단순하여 연산 속도가 매우 빠릅니다. 대용량 데이터를 암호화하는 데 적합합니다.
- **한계:** 키 전달 과정에서 유출될 경우 보안이 완전히 무너지는 '키 교환 문제(Key Exchange Problem)'가 발생합니다.
- **알고리즘 예시:** AES, DES, SEED.

#### 비대칭키 (Asymmetric Key)

- **개념:** 암호화와 복호화에 서로 다른 키 쌍을 사용하는 방식입니다. 이 키 쌍이 바로 뒤에서 설명할 **공개키**와 **비공개키**입니다.
- **특징:** 키 교환 문제를 해결할 수 있으며, 전자서명 등에 활용됩니다. 하지만 대칭키 방식에 비해 연산 복잡도가 매우 높아 속도가 느립니다.
- **알고리즘 예시:** RSA, ECC, Diffie-Hellman.

### 2. 공개키 (Public Key) vs 비공개키 (Private Key)

비대칭키 암호화 알고리즘에서 한 쌍으로 존재하는 두 개의 키를 의미합니다.

- **공개키 (Public Key):** 누구나 알 수 있도록 공개된 키입니다. 주로 데이터를 **암호화**할 때 사용합니다.
- **비공개키 (Private Key):** 본인만 안전하게 보관해야 하는 키입니다. 공개키로 암호화된 데이터를 **복호화**할 때 사용합니다.

> **작동 원리:** A가 B에게 데이터를 보낼 때, B의 **공개키**로 데이터를 암호화합니다. 이 데이터는 B가 가진 **비공개키**로만 풀 수 있으므로 전송 과정에서 가로채더라도 내용을 알 수 없습니다.

### 3. 세션키 (Session Key)

세션키는 특정 통신 세션 동안에만 유효한 **일회성 대칭키**입니다.

- **개념:** 통신 주체들이 데이터를 주고받기 위해 임시로 생성하여 공유하는 키입니다.
- **필요성:** 비대칭키(공개키 방식)는 보안성은 높지만 느리고, 대칭키는 빠르지만 키 전달이 위험합니다. 이를 해결하기 위해 **하이브리드 방식**을 사용합니다.

1. 비대칭키 방식을 사용하여 상대방과 안전하게 **세션키(대칭키)**를 공유합니다.
2. 일단 세션키가 공유되면, 실제 대량의 데이터 전송은 빠른 **대칭키 방식**으로 수행합니다.
3. 통신이 종료되면 해당 세션키는 폐기됩니다.

### 4. 개념 간 비교 요약

| 구분             | 대칭키 (Symmetric)           | 비대칭키 (Asymmetric)                        |
| ---------------- | ---------------------------- | -------------------------------------------- |
| **키의 개수**    | 1개                          | 2개 (공개키, 비공개키)                       |
| **암호/복호 키** | 동일함                       | 다름                                         |
| **속도**         | 빠름                         | 느림 (대칭키 대비 약 100~1000배)             |
| **용도**         | 대용량 데이터 본문 암호화    | 세션키 교환, 전자서명, 인증                  |
| **연관 개념**    | **세션키**가 대칭키의 일종임 | **공개키/비공개키**가 비대칭키의 구성 요소임 |

결론적으로, 현대의 HTTPS(TLS/SSL) 통신 등에서는 **비대칭키(공개키/비공개키)**를 이용해 안전하게 **세션키**를 교환하고, 이후 실제 데이터 통신은 효율적인 **대칭키(세션키)** 방식을 사용하는 구조를 취하고 있습니다.

## 클라이언트는 비밀키를 가지지 않는다??

TLS Handshake 과정에서 **클라이언트는 서버의 비밀키(Private Key)를 절대 가지지 않습니다.** 비밀키는 오직 서버만이 안전하게 보관해야 하며, 이것이 유출되면 해당 서버의 통신은 더 이상 안전하지 않게 됩니다.

클라이언트가 복호화를 수행하는 시점과 그 메커니즘을 단계별로 설명해 드리겠습니다.

### 1. 클라이언트의 복호화: 인증서 검증 단계

클라이언트가 가장 먼저 복호화를 수행하는 데이터는 서버가 보낸 **디지털 서명(Digital Signature)**입니다.

- **서버의 송신:** 서버는 CA(인증 기관)로부터 발급받은 인증서를 보냅니다. 이 인증서에는 서버의 공개키와 함께, CA가 자신의 비밀키로 암호화한 '해시값(서명)'이 들어있습니다.
- **클라이언트의 복호화:** 클라이언트(브라우저)는 이미 신뢰할 수 있는 **CA의 공개키**를 가지고 있습니다. 이 CA 공개키를 사용하여 인증서에 포함된 서명을 복호화합니다.
- **검증:** 복호화된 해시값과 실제 인증서 데이터를 해싱한 값을 비교하여, 인증서가 위조되지 않았음을 확인합니다.

### 2. 키 교환 단계 (RSA 방식 기준)

전통적인 RSA 방식에서는 클라이언트가 데이터를 암호화만 하고, 복호화는 서버의 몫입니다.

1. **클라이언트:** 대칭키로 사용할 **Pre-master Secret**을 생성합니다.
2. **암호화:** 서버로부터 받은 **서버의 공개키**로 이를 암호화하여 서버에 보냅니다.
3. **서버의 복호화:** 서버는 오직 자신만 가진 **비밀키**로 이를 복호화하여 Pre-master Secret을 얻습니다.

- 이 단계에서 클라이언트는 자신이 암호화한 주체이므로 별도의 복호화 과정이 필요 없습니다.

### 3. 실제 데이터 통신 단계 (대칭키 사용)

Handshake가 완료된 후, 클라이언트와 서버는 동일한 **대칭키(Session Key)**를 공유하게 됩니다. 이때부터 클라이언트는 서버가 보낸 데이터를 복호화합니다.

- **서버:** 데이터를 공유된 대칭키로 암호화하여 전송합니다.
- **클라이언트:** 동일한 **대칭키**를 사용하여 서버가 보낸 데이터를 **복호화**합니다.

### 요약

- **서버의 비밀키:** 클라이언트는 절대 가질 수 없으며, 가질 필요도 없습니다.
- **클라이언트의 복호화:** 1. 인증서 검증 시 **CA의 공개키**로 서명을 복호화합니다.

2.  연결 수립 후 **대칭키**를 사용하여 서버의 데이터를 복호화합니다.

비대칭키(공개키 암호화) 방식은 오직 **대칭키를 안전하게 전달하기 위한 수단**으로만 사용된다고 이해하시면 정확합니다.

## HTTPS

**HTTPS(Hypertext Transfer Protocol Secure)**는 HTTP의 보안 버전으로, 클라이언트와 서버 간의 통신을 암호화하여 데이터의 무결성과 기밀성을 보장하는 프로토콜입니다. OSI 7 계층 관점에서 보면, 애플리케이션 계층의 HTTP와 전송 계층의 TCP 사이에 **TLS(Transport Layer Security)** 또는 과거의 SSL 계층을 추가하여 동작합니다.

### 1. 주요 보안 메커니즘

HTTPS는 다음 세 가지 핵심 보안 요소를 제공합니다.

- **기밀성(Confidentiality):** 제3자가 통신 내용을 훔쳐보더라도 암호화되어 있어 내용을 알 수 없습니다.
- **무결성(Integrity):** 데이터가 전송되는 도중 변조되지 않았음을 보장합니다.
- **인증(Authentication):** 접속하려는 서버가 신뢰할 수 있는 대상인지 증명합니다.

### 2. TLS Handshake 프로세스

HTTPS 연결이 수립될 때, 클라이언트와 서버는 **TLS Handshake** 과정을 통해 암호화 키를 교환합니다.

1. **Client Hello:** 클라이언트가 지원하는 암호화 방식(Cipher Suite)과 무작위 문자열을 서버에 보냅니다.
2. **Server Hello & Certificate:** 서버는 암호화 방식을 선택하고, CA(Certificate Authority)로부터 발급받은 **SSL 인증서**를 클라이언트에 전달합니다. 이 인증서에는 서버의 **공개키(Public Key)**가 포함되어 있습니다.
3. **Authentication:** 클라이언트는 브라우저에 내장된 CA 리스트를 통해 인증서의 유효성을 검증합니다.
4. **Key Exchange:** 클라이언트는 무작위 문자열을 조합해 **대칭키(Session Key)**를 생성하고, 이를 서버의 공개키로 암호화하여 전송합니다.
5. **Session Established:** 서버는 자신의 비밀키(Private Key)로 이를 복호화하여 대칭키를 획득합니다. 이후 모든 데이터 통신은 이 대칭키를 사용해 암호화됩니다.

### 3. 암호화 방식의 특징

HTTPS는 효율적인 데이터 전송을 위해 **비대칭키**와 **대칭키** 방식을 혼합하여 사용합니다.

| 방식                      | 사용 시점             | 특징                                                        |
| ------------------------- | --------------------- | ----------------------------------------------------------- |
| **비대칭키 (Asymmetric)** | 핸드셰이크 단계       | 공개키와 비밀키를 사용하여 안전하게 대칭키를 공유함         |
| **대칭키 (Symmetric)**    | 실제 데이터 전송 단계 | 공유된 세션 키를 사용하여 고속으로 데이터를 암호화/복호화함 |

### 4. 사용 포트

- **HTTP:** 80번 포트 (평문 전송)
- **HTTPS:** 443번 포트 (암호화 전송)

HTTPS는 현대 웹 표준의 필수 요소이며, 검색 엔진 최적화(SEO) 및 브라우저의 보안 경고 방지를 위해서도 반드시 적용해야 합니다.

## 1. 회원가입: 사용자 정보의 저장

사용자가 회원가입 버튼을 누르면 데이터는 아래와 같은 경로로 흐릅니다.

- **프론트엔드 (`register/page.tsx`)**: 사용자가 입력한 데이터를 `api.ts`로 넘깁니다.
- **백엔드 API (`app.py`)**: `generate_password_hash`를 통해 비밀번호를 암호화한 뒤 DB에 저장합니다.

```python
# backend/app.py
password_hash = generate_password_hash(password) # 비밀번호 암호화
query_db(
    "INSERT INTO users (username, email, password_hash) VALUES (%s, %s, %s)",
    (username, email, password_hash),
)

```

## 2. 로그인: 세션의 생성

로그인이 성공하면 서버는 해당 사용자를 식별할 수 있는 정보를 **서버 메모리(세션)**에 저장합니다.

- **백엔드 세션 저장**: `session` 객체에 유저 ID를 기록합니다. 이때 Flask는 내부적으로 고유한 **세션 ID**를 생성하여 브라우저에 쿠키 형태로 전달합니다.

```python
# backend/app.py
@app.route("/api/auth/login", methods=["POST"])
def login():
    # ... 유저 확인 로직 ...
    session["user_id"] = user["id"] # 서버 측 세션에 저장
    session["username"] = user["username"]
    return jsonify({"message": "로그인 성공", "user": ...})

```

## 3. 로그인 유지: 쿠키와 세션의 결합 (Credentials)

사용자가 로그인 후 다른 페이지로 이동하거나 게시글을 쓸 때, 매번 아이디/비밀번호를 입력하지 않아도 되는 이유는 이 세션 방식 덕분입니다.

- **프론트엔드 요청 (`api.ts`)**: `credentials: "include"` 설정이 핵심입니다. 이 설정 덕분에 브라우저는 매 요청마다 서버가 준 세션 쿠키를 자동으로 함께 보냅니다.

```typescript
// frontend/src/lib/api.ts
async function request<T>(
  endpoint: string,
  options: RequestInit = {},
): Promise<T> {
  const res = await fetch(`${API_BASE}${endpoint}`, {
    headers: { "Content-Type": "application/json" },
    credentials: "include", // 브라우저에 저장된 세션 쿠키를 서버로 자동 전송
    ...options,
  });
  return res.json();
}
```

## 4. 인증 확인: 서버에서의 사용자 식별

서버는 들어온 요청에 포함된 쿠키(세션 ID)를 보고 "아, 이 사람은 아까 로그인한 user_id 1번 이종윤님이구나"라고 판단합니다.

- **권한 검사 (`app.py`)**: 게시글 작성이나 내 정보 조회 시 세션 값이 있는지 확인합니다.

```python
# backend/app.py
@app.route("/api/auth/me", methods=["GET"])
def me():
    user_id = session.get("user_id") # 쿠키를 통해 서버 세션에서 ID를 가져옴
    if user_id is None:
        return jsonify({"error": "로그인이 필요합니다."}), 401
    # ... 이후 유저 정보 반환

```
