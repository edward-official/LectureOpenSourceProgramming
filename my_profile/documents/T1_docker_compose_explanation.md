# docker-compose.yml 파일 구조 및 설명

이 문서는 프로젝트 최상단에 위치한 `docker-compose.yml` 파일의 각 서비스(컨테이너)가 어떻게 설정되어 있고 서로 어떻게 통신하는지 설명합니다.

## 전체 개요

Docker Compose는 여러 개의 컨테이너로 이루어진 애플리케이션을 하나의 명령어로 정의하고 실행할 수 있게 해주는 도구입니다.
본 프로젝트에서는 `db`(데이터베이스), `backend`(API 서버), `frontend`(클라이언트) 총 3개의 서비스로 구성되어 있습니다.

```yaml
services:
  db: ...
  backend: ...
  frontend: ...
```

## 도커(Docker)란?

도커(Docker)는 애플리케이션과 그 실행에 필요한 모든 환경을 하나로 묶어 어디서나 동일하게 실행할 수 있게 돕는 컨테이너 가상화 플랫폼입니다.

### 1. 도커 컨테이너의 핵심 개념

- **패키징**: 코드, 라이브러리, 설정 파일을 '컨테이너'라는 표준화된 상자에 담습니다.
- **격리 및 공유**: 각 컨테이너는 독립된 공간에서 실행되지만, 호스트의 OS 커널을 공유하여 자원을 효율적으로 사용합니다.
- **일관성**: "내 컴퓨터에서는 되는데 서버에서는 안 돼"라는 문제 없이, 어떤 환경(리눅스, 맥, 윈도우)에서든 동일한 동작을 보장합니다.

### 2. 커널 공유와 운영체제별 차이

- **리눅스**: 호스트의 리눅스 커널을 직접 사용하여 가장 빠르고 가볍게 작동합니다.
- **맥 & 윈도우**: 리눅스 커널이 없으므로, 도커가 내부적으로 **경량 리눅스 가상 머신(VM)**을 띄워 그 안의 커널을 컨테이너들이 공유하게 만듭니다.

## 포트 연결(Port Mapping)을 하는 이유

도커 환경에서 **포트 연결(Port Mapping)**을 하는 이유는 크게 세 가지 핵심적인 목적이 있습니다.

### 1. 격리된 컨테이너의 '문'을 열어주기 위해

도커 컨테이너는 기본적으로 호스트 컴퓨터와 분리된 독립적인 가상 네트워크 안에서 실행됩니다.

- 컨테이너 내부에서 아무리 웹 서버(예: 3000번 포트)를 돌리고 있어도, 외부(호스트 컴퓨터의 브라우저)에서는 컨테이너 내부로 들어가는 경로를 알 수 없습니다.
- **포트 연결**은 호스트의 특정 포트와 컨테이너의 특정 포트를 이어주는 **전용 통로**를 만드는 작업입니다.

### 2. 서비스 접근성 확보 (외부 통신)

사용자가 실제로 서비스를 이용할 수 있도록 하기 위함입니다.

- **프런트엔드**: 사용자가 브라우저 주소창에 `localhost:3000`을 입력했을 때, 호스트의 3000번 포트가 신호를 받아 컨테이너 내부의 Next.js 서버로 전달해 줍니다.
- **백엔드**: 프런트엔드 앱이나 API 테스트 도구가 `localhost:5001`로 요청을 보내면, 이 연결을 통해 컨테이너 안의 Flask 앱이 요청을 수신합니다.
- **데이터베이스**: 개발자가 외부 GUI 툴(예: DBeaver, pgAdmin)로 DB를 관리하고 싶을 때, 호스트의 5432 포트를 통해 컨테이너 안의 PostgreSQL에 접속할 수 있습니다.

### 3. 포트 충돌 방지 및 관리

호스트 컴퓨터에서는 여러 개의 컨테이너가 동시에 돌아갈 수 있습니다.

- 만약 두 개의 서로 다른 프로젝트가 모두 컨테이너 내부에서 `8080` 포트를 사용한다면 충돌이 날 수 있습니다.
- 이때 포트 연결을 통해 **A 프로젝트는 `8081:8080`**, **B 프로젝트는 `8082:8080`**으로 설정하면, 호스트 입장에서는 서로 다른 포트로 두 서비스를 구분하여 관리할 수 있습니다.

> **요약하자면**: 포트 연결은 **"외부(호스트/사용자)에서 컨테이너 내부의 서비스에 접속할 수 있는 유일한 통로를 정의하는 것"**이라고 이해하시면 됩니다.

## 1. db (PostgreSQL 데이터베이스)

- **image**: `postgres:16-alpine`
  경량화된 alpine 리눅스 기반의 PostgreSQL 16 버전을 사용합니다.
- **environment**:
  데이터베이스 초기 설정(DB 이름, 사용자명, 비밀번호)을 환경 변수로 주입합니다.
- **ports**: `"5432:5432"`
  호스트 컴퓨터의 5432 포트와 컨테이너 내부의 5432 포트를 연결합니다. 외부(DBeaver 등)에서 직접 DB에 접속할 수 있게 해줍니다.
- **volumes**: `pgdata:/var/lib/postgresql/data`
  컨테이너가 종료되거나 삭제되어도 데이터가 날아가지 않도록, 호스트 머신에 볼륨(`pgdata`)을 만들어 데이터를 영구 저장합니다.
- **healthcheck**:
  데이터베이스가 컨테이너로 띄워지고 나서, 실제로 연결을 받을 수 있는 상태인지 검사합니다.

## 2. backend (Flask 서버)

- **build**: `./backend`
  `backend` 폴더 안에 있는 `Dockerfile`을 이용해 이미지를 직접 빌드합니다.
- **ports**: `"5001:5001"`
  호스트 컴퓨터의 5001 포트와 컨테이너의 5001 포트를 연결합니다. (로컬에서 백엔드 API만 따로 테스트해 볼 수 있습니다)
- **environment**:
  Flask 앱 실행 파일(`app.py`), DB 접속 정보 등 백엔드가 작동하는 데 필요한 환경 변수들을 설정합니다. 여기서 `DB_HOST: db`로 설정되어 있는데, 이는 Docker 네트워크 상에서 컨테이너 이름(`db`)만으로 데이터베이스와 통신할 수 있음을 의미합니다.
- **volumes**: `./backend:/app`
  내 컴퓨터의 `./backend` 폴더와 컨테이너 내부의 `/app` 폴더를 동기화합니다. 즉 서버를 껐다 켜지 않아도 코드를 수정하면 실시간으로 반영됩니다(Hot Reload).
- **depends_on**: `db`
  데이터베이스 컨테이너가 먼저 실행되고 정상 상태(`service_healthy`)가 된 이후에 백엔드 서버를 켭니다.

## 3. frontend (Next.js 애플리케이션)

- **build**: `./frontend`
  `frontend` 폴더 안에 있는 `Dockerfile`을 이용해 이미지를 직접 빌드합니다.
- **ports**: `"3000:3000"`
  호스트 컴퓨터의 3000 포트와 웹 브라우저가 접속할 수 있는 컨테이너의 포트를 연결합니다.
- **volumes**:
  - `./frontend:/app` : 로컬의 코드를 컨테이너에 동기화합니다.
  - `/app/node_modules`, `/app/.next` : 컨테이너 내부에 생성된 의존성과 빌드 산출물을 내 컴퓨터의 파일들로 덮어쓰지 않도록(충돌 방지) 별도 볼륨으로 잡습니다.
- **depends_on**: `backend`
  백엔드 컨테이너가 먼저 실행된 후에 프론트엔드를 실행합니다.

## 상태 확인(Healthcheck)과 서비스 간 연동

도커에서 **상태 확인(Healthcheck)**은 컨테이너 내부의 프로세스가 단순히 실행 중인 것을 넘어, 실제로 서비스를 제공할 수 있는 **'건강한 상태'**인지 주기적으로 검사하는 기능입니다.

### 1. 왜 필요한가요?

일반적으로 도커는 컨테이너 안의 프로세스(예: PostgreSQL)가 종료되지 않으면 '실행 중(Running)'으로 표시합니다. 하지만 프로세스가 떠 있더라도 초기화 중이거나 오류로 인해 응답을 못 하는 상황일 수 있습니다. 상태 확인은 이런 상황을 감지하여 서비스의 신뢰성을 높여줍니다.

### 2. 구체적인 동작 원리 (db 서비스 기준)

파일에 정의된 `db` 서비스의 설정을 보면 다음과 같이 작동합니다:

- **검사 명령 (`test`)**: `pg_isready -U postgres` 명령을 실행하여 데이터베이스가 접속을 받을 준비가 되었는지 확인합니다.
- **검사 주기 (`interval: 5s`)**: 5초마다 한 번씩 위의 검사 명령을 반복합니다.
- **응답 제한 (`timeout: 5s`)**: 검사 명령을 내린 후 5초 안에 응답이 없으면 해당 회차는 실패로 간주합니다.
- **재시도 횟수 (`retries: 5`)**: 연속으로 5번 실패하면 해당 컨테이너의 상태를 'unhealthy(비정상)'로 변경합니다.

### 3. 서비스 간의 연동 (의존성 관리)

상태 확인의 가장 큰 장점은 서비스 간의 실행 순서를 정교하게 제어할 수 있다는 점입니다.

- **설정 내용**: `backend` 서비스는 `depends_on` 설정에서 `db` 서비스의 `condition: service_healthy`를 요구합니다.
- **효과**: `db` 컨테이너가 단순히 '시작'된 직후가 아니라, 상태 확인을 통해 **실제로 데이터베이스 접속이 가능한 상태**가 되었을 때 비로소 `backend` 컨테이너가 실행됩니다. 이를 통해 백엔드가 DB 연결에 실패하여 비정상적으로 종료되는 문제를 방지할 수 있습니다.

## 도커 볼륨(Volume): 데이터 보존과 실시간 코드 반영

`docker-compose.yml` 파일에서 `db` 서비스와 `frontend`/`backend` 서비스가 볼륨을 사용하는 방식은 그 목적과 동작 원리에서 뚜렷한 차이가 있습니다. 

### 1. db 서비스: 데이터 보존을 위한 '명명된 볼륨 (Named Volume)'
`db` 서비스에서 사용하는 방식은 데이터를 안전하게 보관하는 데 초점이 맞춰져 있습니다.
- **설정 방식**: `pgdata:/var/lib/postgresql/data`
- **동작 원리**: 도커가 관리하는 별도의 저장 공간(`pgdata`)을 만들고, 이를 컨테이너 내부의 데이터 저장 경로와 연결합니다.
- **사용 이유**: 컨테이너는 삭제되면 내부 데이터가 모두 사라지는 특성이 있습니다. 하지만 이렇게 '명명된 볼륨'을 사용하면 컨테이너를 지우거나 새로 만들어도 실제 데이터베이스 파일은 `pgdata`라는 공간에 남아있으므로 데이터 유실을 방지할 수 있습니다.

### 2. frontend / backend 서비스: 개발 편의를 위한 '바인드 마운트 (Bind Mount)'
`frontend`와 `backend` 서비스에서 사용하는 방식은 호스트 컴퓨터의 파일을 컨테이너와 동기화하는 데 초점이 맞춰져 있습니다.
- **설정 방식**: `./frontend:/app` 또는 `./backend:/app`
- **동작 원리**: 내 컴퓨터(호스트)의 특정 디렉토리 경로를 컨테이너 내부 경로와 직접 연결합니다.
- **사용 이유**: 소스 코드를 수정할 때마다 매번 이미지를 다시 빌드할 필요가 없기 때문입니다. 내 컴퓨터에서 코드를 수정하고 저장하면 그 변경 사항이 컨테이너 내부의 `/app` 폴더에도 즉시 반영되어, 실행 중인 애플리케이션에 바로 적용됩니다.

### 3. 추가적인 차이: 익명 볼륨 (Anonymous Volume)
`frontend` 서비스 설정을 보면 조금 특이한 형태의 볼륨 설정이 더 포함되어 있습니다.
- **설정 방식**: `- /app/node_modules`, `- /app/.next`
- **동작 원리**: 이는 호스트의 파일이 컨테이너 내부의 해당 폴더를 덮어쓰지 못하도록 보호하는 설정입니다.
- **사용 이유**: `node_modules` 같은 라이브러리 폴더는 컨테이너 환경 내에서 설치된 것을 그대로 사용해야 안전합니다. 바인드 마운트로 인해 호스트의 설정이 컨테이너 내부의 환경을 오염시키는 것을 막아주는 보호막 역할을 합니다.

---
```yaml
volumes:
  pgdata:
```
하단에 선언된 `volumes: pgdata:` 부분은 위에서 설명한 `db` 서비스의 '명명된 볼륨(Named Volume)'을 전역으로 선언하여 도커가 관리하도록 하는 필수 구문입니다.
